## 1. Project Scope

`tradingview-mcp-go` is a Go implementation of a TradingView Desktop MCP server and CLI.

Its responsibility is:

- connect to TradingView Desktop through Chrome DevTools Protocol / CDP;
- expose MCP tools over stdio JSON-RPC;
- expose equivalent CLI commands;
- launch/discover TradingView Desktop where possible;
- capture TradingView state, chart data, screenshots, Pine editor/script state, replay state, alerts, panes, tabs, layouts, and UI interactions;
- preserve compatibility with the original Node.js implementation where applicable.

Do **not** add unrelated systems.

Out of scope for this repository:

- HTS decision engine;
- Tinkoff Invest API;
- order execution;
- trade planning;
- Telegram control bot;
- multi-user platform;
- AI provider routing;
- backtesting engine;
- risk engine;
- broker integration;
- automatic trading.

If a task appears to require those systems, document the boundary and keep this repository focused on TradingView MCP functionality only.

## 3. Core Rules

### 3.1 Preserve MCP compatibility

Do not rename tools, remove tools, change argument names, or change response shape unless there is a proven bug and the change is documented.

Every MCP tool must keep the general response convention:

```json
{
  "success": true
}
```

or:

```json
{
  "success": false,
  "error": "..."
}
```

For MCP tool execution failures, return MCP `CallToolResult` with `isError: true` where appropriate, not a JSON-RPC protocol error, unless the JSON-RPC request itself is invalid.

### 3.2 No fake success

Do not hide incomplete functionality behind `success: true`.

If a tool cannot obtain real data from TradingView, it must return one of:

- `success: false` with a clear reason;
- `success: true` with explicit fields showing that data is unavailable, partial, or estimated;
- a documented fallback mode.

Never silently return canvas coordinates as if they were real indicator values.

### 3.3 Do not treat canvas coordinates as indicator values

Canvas/Y coordinates are visual rendering coordinates, not financial data.

If the code obtains values from canvas hit-testing, pixel probing, or DOM layout positions, clearly mark them as visual coordinates.

Real indicator values must come from one of:

- TradingView exposed data model;
- data window / study values if accessible;
- Pine editor/strategy/report APIs if accessible;
- explicit text/table/label output generated by the indicator;
- OHLCV-based calculation only if a tool explicitly documents it as local calculation.

### 3.4 Prefer small, testable changes

Make one focused change at a time.

After each meaningful change, run the smallest relevant test set and update docs/results if behavior changed.

Do not perform broad rewrites unless a narrow fix is impossible.

### 3.5 Do not add heavy dependencies casually

The project is intentionally lightweight.

Before adding a dependency, prefer:

- Go standard library;
- existing project packages;
- small local helper functions.

Any new dependency must be justified in `CHANGELOG.md` or the relevant issue note.

## 5. Required Workflow

Before editing:

1. Inspect the relevant files.
2. Identify the exact tool/command affected.
3. Check whether there is an existing audit or smoke-test result for it.
4. Preserve existing public behavior unless fixing a documented bug.

During editing:

1. Keep changes minimal.
2. Prefer explicit error messages.
3. Add comments only where they clarify non-obvious CDP or TradingView behavior.
4. Avoid speculative code.

After editing:

1. Run `go test ./...` when possible.
2. Run `go vet ./...` when possible.
3. Run targeted CLI/MCP smoke checks when the change affects runtime behavior.
4. Update `CHANGELOG.md`, `TODO_codex.md`, or the appropriate result document.
5. Record whether the issue is fixed, partially fixed, blocked by TradingView, or requires manual verification.

## 6. Testing Expectations

### 6.1 Unit tests

Add unit tests for deterministic logic:

- path discovery;
- symbol normalization;
- JSON-RPC parsing/formatting;
- MCP registry;
- filename normalization;
- CLI argument parsing;
- response shaping;
- error mapping.

### 6.2 Golden tests

Maintain golden tests for MCP basics:

- `initialize`;
- `notifications/initialized`;
- `ping`;
- `tools/list`;
- `tools/call` success;
- `tools/call` unknown tool;
- invalid JSON;
- invalid method;
- invalid params.

### 6.3 Smoke tests

Smoke tests that require TradingView Desktop must be clearly separated from normal unit tests.

They should verify at least:

- CDP port availability;
- `tv status`;
- `tv discover`;
- `chart_get_state`;
- `quote_get`;
- `data_get_ohlcv`;
- `capture_screenshot`;
- `symbol_search`;
- one Pine-related command;
- one replay-related command, if TradingView UI state allows it.

Do not make normal CI fail just because TradingView Desktop is not running.

## 7. MCP input reader robustness

Inspect current stdin reader.

If using `bufio.Scanner`, verify message size safety.

Acceptance criteria:

- large `tools/call` requests do not fail at 64 KiB;
- either increase scanner buffer or replace with a line reader/decoder;
- invalid/malformed messages still return correct JSON-RPC errors.

## 9. CLI Rules

The `tv` CLI must be useful for debugging without MCP clients.

For each CLI command:

- stdout should be machine-readable when JSON mode is expected;
- stderr should be used for human diagnostics if needed;
- exit code must reflect failure;
- avoid printing unrelated logs into JSON output;
- keep command names stable.

For stream commands:

- JSONL output is acceptable by design;
- each line must be valid JSON;
- errors should also be JSONL records where possible.

---

## 10. MCP Stdio Rules

The MCP server must not write debug logs to stdout.

Stdout is reserved for JSON-RPC responses/notifications.

Use stderr for diagnostics.

Never print banners, progress lines, or human-readable logs to stdout in MCP mode.

---

## 11. Documentation Rules

When a behavior is verified, document:

- date of verification;
- OS;
- TradingView Desktop version if known;
- symbol used;
- command/tool used;
- result file path;
- whether the result is full, partial, or blocked.

When a behavior cannot be supported, document it honestly.

Do not claim TradingView limitations are fixed unless there is a real reproducible test.
